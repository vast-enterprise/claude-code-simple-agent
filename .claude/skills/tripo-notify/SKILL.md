### 动态接收人：提出人

除了固定所有者，每条通知的接收人还必须包含**对应记录的提出人**。提出人按流程类型从不同字段读取：

| 流程类型 | 提出人字段 | 读取方式 |
|---|---|---|
| Requirement（R1-R4）| 产品需求池 / 技术需求池的 `需求Owner`（`fldeFXWrBP`）| `lark-cli base record-get` 按 `record_id` 读取 |
| Bugfix（B1-B2）| Bug 表的**备注字段**（`fldemHmVhH`，scrum-master 录入时用飞书原生 `@` 人语法标注） → fallback 到 `创建人`（`fld1mMy65N`）| `record-get` 读取备注字段，从富文本中提取 `@` 结构的 open_id；提取不到时回落到创建人字段 |
| 执行表场景 | 执行表本身不记录提出人，也没有 link 字段直指来源需求池 | **暂不支持自动回查**。触发通知时向用户询问来源记录所在表（产品需求池 / 技术需求池）和 record_id，再按 Requirement 规则读 `需求Owner` |

**为什么 Bug 表要用备注 fallback**：Bug 表的 `创建人` 是 `created_by` 系统字段，在 scrum-master 代录入场景下会填成机器人账号。所以 `tripo-bugfix` step 2 要求录入时在备注字段**用飞书原生 `@` 人语法**显式记录真实提出人（读出来就是富文本结构，直接取 open_id，无需字符串解析）。录入规则详见 `tripo-tables` 的「Bug 表特别规则」。

## 通知渠道规则

每条通知的最终接收人 = { 交互用户 } ∪ { 提出人 } ∪ { 所有者 }（去重）：

- **私聊交互**：
  - 私聊发给"交互用户"
  - 若"提出人"不同于交互用户 → 额外私聊"提出人"
  - 若"所有者"不同于前两者 → 额外私聊"所有者"
- **群聊交互**（在群里收到的 bug 报告 / 需求）：
  - 在群里**回复原消息**（反@报告人）
  - 若"提出人"不同于群里报告人 → 额外私聊"提出人"
  - 额外私聊"所有者"（除非重合）

任意两方重合时不重复发。

## 通知协议

```
1. 完成步骤工作（输出文档 / 创建 PR / 定位根因）
2. 发送飞书通知（→ lark-im skill）
3. 暂停，等待用户确认
4. 用户确认后继续下一步
```

注意事项：
- **业务约束：必须用应用身份发送**（不用用户身份冒充个人发通知），身份切换细节 → lark-shared skill
- **业务约束：实际发送前必须预览一次**（避免 at 错人、模板占位符未填），预览机制 → lark-shared skill
- 通知发送失败时提示用户，不阻塞流程

## 通知节点

> **所有节点的接收人** = { 交互用户 } ∪ { 提出人 } ∪ { 所有者 }（见上方"通知渠道规则"），下表只列触发时机和消息模板。

### Requirement 流程

| 节点 | 步骤 | 触发时机 | 消息模板 |
|------|------|---------|---------|
| R1 | 3 | review.md 输出后 | `[需求评审完成] 需求: <名称> / 文档: <Wiki链接> / 操作: 请确认容量` |
| R2 | 5 | technical-solution.md + prototype/index.html 都输出后（原型不适用时仅方案） | `[技术评审完成] 需求: <名称> / 方案: <Wiki链接> / 原型: <prototype/index.html 链接或"无 UI 变更"> / 操作: 请确认方案与原型` |
| R3 | 8 | Code Review + 测试通过后 | `[PR 已完成验证] 需求: <名称> / PR: <链接> / 文档: CR+测试计划+报告 Wiki链接 / 操作: 请 review 并合并` |
| R4 | 10 | 验收通过，准备上线 | `[准备上线] 需求: <名称> / 操作: 请确认发布` |
