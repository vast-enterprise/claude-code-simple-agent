# 需求澄清与任务规划方法论

## 总则：澄清先于评审，评审先于拆分

需求不清楚就直接写文档，等于把模糊性打包递给下一个岗位。澄清的目的是**把所有会在后续步骤被追问的问题，在评审阶段一次问透**——让后续的技术评审、开发、测试都能基于同一份明确需求运转。

澄清完才写评审文档，评审文档确认完才做任务拆分——这个顺序不能反。

## 12 澄清维度

每个新需求必须过一遍 12 维度，缺一不可。缺失的维度要么调用 AskUserQuestion 问清楚，要么在评审文档里显式标注"未澄清，需用户补充"——不能默认脑补。

| # | 维度 | 要回答的问题 | 缺失后果 |
|---|------|-------------|---------|
| 1 | **用户** | 谁用这个功能？B 端/C 端/内部运营？一次用多少人？ | 工作量估错、权限设计错 |
| 2 | **场景** | 在什么业务场景触发？日常高频还是偶发？ | 性能与资源规划错位 |
| 3 | **目标** | 做成之后衡量成功的指标是什么？ | 做完发现"不是用户要的" |
| 4 | **核心功能** | 必须要做的最小集是什么？ | 范围蔓延 |
| 5 | **边界** | 哪些看似相关但**不做**？ | 做多了白做，做少了返工 |
| 6 | **验收标准** | 怎样叫"做完了"？可测试吗？ | 验收时各说各话 |
| 7 | **依赖** | 依赖哪些上下游、第三方、现有数据？ | 开工后才发现卡脖子 |
| 8 | **数据** | 要读哪些表 / 写哪些表 / 数据格式？ | 数据迁移 / 回填阶段崩盘 |
| 9 | **权限** | 谁能看 / 谁能改 / 需要登录态吗？ | 安全漏洞或权限过松 |
| 10 | **异常** | 失败了怎么办？用户看到什么？ | 线上故障无 fallback |
| 11 | **合规与真实性** | 有没有外部政策冲突（Google 结构化数据、App Store、第三方 TOS）？有没有伪造数据风险（虚假评分、捏造统计）？ | 平台封禁、法律风险 |
| 12 | **时间压力** | 什么时候上线？紧急程度？ | 方案选型与范围错配 |

**澄清的提问姿态**：不是一次性抛 12 个问题（那是问卷调查），而是基于已知信息识别**最关键的未知维度**，每轮用 AskUserQuestion 聚焦 2-3 个问题，逐步收敛。

> 详见 [references/clarification-matrix.md](references/clarification-matrix.md) —— 每个维度的问题模板与常见反模式。

## PRD 文档结构（review.md 内容标准）

评审文档就是 PRD（Product Requirement Document）。结构固定，便于下游岗位快速定位信息：

```markdown
# <需求名>

## 需求来源
- ID / 表格链接
- 需求 Owner（默认 = 提出人，可在确认时修正）/ 研发 Owner
- 来源文档（飞书文档 / 用户反馈 / 竞品分析）
- 提出日期

## 业务背景
- 目前的问题（数据 / 用户反馈）
- 做这个需求想解决什么

## 需求范围

### 核心功能（必做）
- 功能 1：<一句话描述>
- 功能 2：<一句话描述>

### 边界（不做）
- 不做 X，因为 <原因>

### 用户场景
- 场景 1：<谁 / 在哪 / 做什么 / 期望什么>

## 验收标准
- [ ] 可测试的验收条件 1
- [ ] 可测试的验收条件 2
（每条必须可观测，不要"体验良好"这种主观描述）

## 技术约束
- 依赖服务 / 表 / API
- 数据格式约束
- 性能要求（如有）

## 合规风险（如有）
| 风险点 | 来源政策 | 违规后果 | 建议 |
|-------|---------|---------|------|
| ... | ... | ... | 保留 / 移除 / 替代方案 |

## 问题与风险
- 未澄清问题：<用户需要补充的信息>
- 技术风险：<技术评审前的初步风险点>
- 资源风险：<依赖方是否 ready>

## 评审结论
- 进入技术评审：是 / 否
- 需要补充澄清：<列出未澄清项>
- 建议优先级：<P0/P1/P2>
```

> 详见 [references/prd-template.md](references/prd-template.md) —— 各段落填充原则与常见问题。

## 任务拆分原则

澄清 + PRD 完成后才拆任务。拆任务不是列 todo，是把"一个需求"切成**独立可验收的子交付**。

### 拆分标准

1. **按"交付"而非"步骤"拆**
   - ❌ "查数据 / 写接口 / 画页面"（按工种切，子任务没法独立验收）
   - ✅ "列表页查询 / 详情页展示 / 编辑提交"（按用户感知的功能切）

2. **每个子任务必须能独立 demo**
   - 拆完后每个任务做完可以给用户看到某种可感知的变化
   - 如果一个子任务做完用户什么都看不见，说明拆得太细或跟别的任务耦合了

3. **识别关键路径**
   - 哪些任务必须按顺序做？（前置依赖）
   - 哪些任务可以并行？（独立交付）
   - 关键路径上的任务延期 = 整个需求延期

4. **工时估算按"P50 / P90"**
   - P50：顺利情况的工时
   - P90：遇到坑 / 需要返工的悲观工时
   - 关键路径任务的 P90 相加 = 整个需求的工期红线

### 拆分反模式

| 反模式 | 问题 | 纠正 |
|-------|------|------|
| 按工种切（前端 / 后端 / 测试） | 子任务无法独立验收 | 按用户可感知的功能切 |
| 粒度过细（10 分钟一个任务） | 管理成本 > 执行成本 | 合并到 0.5-2 天粒度 |
| 粒度过粗（"做完整个需求"） | 无法追踪进度 | 拆到可在 2 天内验收 |
| 漏掉测试 / 文档 | 后面返工补 | 每个功能任务自带测试子任务 |
| 漏掉迁移 / 回填 | 上线时卡住 | 涉及数据变更的任务必带迁移子任务 |

## 风险识别清单

需求评审阶段必过的风险扫查。每一条命中就写进 PRD 的"合规风险"或"问题与风险"段。

### 合规与真实性（强制）
- [ ] 是否涉及**外部平台政策**？（Google 结构化数据 / App Store 审核 / 第三方 API TOS）
- [ ] 是否有**数据真实性风险**？（伪造评分、虚构用户、捏造统计）
- [ ] 是否涉及**用户隐私数据**？（手机号 / 邮箱 / 地址 / 行为日志）
- [ ] 是否涉及**支付 / 财务数据**？（特殊审计要求）

### 技术风险
- [ ] 有没有引入**新依赖**？（新库 / 新服务 / 新中间件）
- [ ] 有没有**跨仓库改动**？（需要协调多个 PR）
- [ ] 有没有**破坏性变更**？（schema 变更 / API 签名变更 / 数据迁移）
- [ ] 有没有**性能敏感点**？（N+1 / 大表扫描 / 高频调用）

### 资源与协作
- [ ] 需要**其他团队配合**吗？（设计 / 运维 / 数据）
- [ ] 依赖的数据**现在存在吗**？（还是需要先建）
- [ ] 有没有**时间冲突**？（发车窗口 / 封板期 / 合规审查期）

### 验收风险
- [ ] 验收标准**可测试**吗？（"体验良好"不叫可测试）
- [ ] 有没有**灰度 / 回滚方案**？（上线后不符合预期怎么办）
- [ ] 有没有**数据回滚能力**？（写入的数据能撤销吗）

> 详见 [references/risk-checklist.md](references/risk-checklist.md) —— 每条风险的判断方法与处理建议。

## 输出格式

不论是澄清阶段还是评审阶段，输出都需要：

1. **已澄清清单**：12 维度里已明确的，一句话列出来
2. **待澄清清单**：还模糊的维度 + 具体问题（用 AskUserQuestion 问）
3. **识别的风险**：按合规 / 技术 / 资源 / 验收四类列出
4. **下一步**：进入评审文档输出 / 继续澄清 / 进入技术方案
