# PRD 模板：各段落填充原则

> SKILL.md 的展开篇。给每个段落写出"好 vs 差"的对比示例。

## 段落 1：需求来源

**目标**：让下游岗位 5 秒内知道这个需求从哪来、找谁问细节。

**最小字段**：
- 需求 ID（飞书表 record-id / 内部编号）
- 需求 Owner（默认 = 提出人，可在确认时修正；对应飞书表 `需求Owner` 字段；R1-R4 通知接收人之一）
- 研发 Owner
- 来源文档链接（PRD / 飞书文档 / 竞品分析）
- 提出日期

**好的例子**：
```
- ID: recXYZ123
- 需求 Owner: 张三
- 研发 Owner: 李四
- 来源: https://xxx.feishu.cn/docs/...
- 提出日期: 2026-04-18
```

**差的例子**：
```
这个是产品要做的，找小明。  ← 没有链接，没有日期
```

## 段落 2：业务背景

**目标**：回答"为什么要做"——让实施者理解动机，遇到权衡时能自己判断。

**最小字段**：
- 当前的问题或机会
- 支持的数据（用户反馈 / 运营数据 / 竞品观察）

**好的例子**：
```
当前问题：博客详情页的内链跳转深度不够，用户看完一篇文章后
没有明显的下一步入口，相关阅读点击率只有 3%（数据来自 GA，
2026-Q1 平均值）。竞品 X 在同场景下的相关阅读点击率是 18%。
```

**差的例子**：
```
产品想优化博客详情页的体验。  ← 没说具体问题，实施时无法判断取舍
```

## 段落 3：需求范围

**目标**：明确做什么、不做什么——范围蔓延和范围漏项都靠这段拦住。

### 核心功能（必做）

每条一句话，可以映射到一个验收标准。

**好的例子**：
```
- 功能 1：博客详情页底部新增"相关阅读"模块，展示 4 篇相似文章
- 功能 2：相似度算法：按 tag 重合度 + 发布时间衰减（近 60 天优先）
- 功能 3：点击相关阅读文章跳转到对应详情页，并埋点上报点击事件
```

**差的例子**：
```
- 做一个相关阅读功能  ← 太笼统，实施者要自己脑补
```

### 边界（不做）

**好的例子**：
```
- 不做相关视频（只做文章之间的关联）
- 不做个性化推荐（不用用户画像，只用文章相似度）
- 不改现有详情页的其他模块
```

**差的例子**：
```
略（不写边界段 = 默认什么都做）
```

### 用户场景

**好的例子**：
```
- 场景 1：注册用户看完一篇 SEO 文章后 → 看到"相关阅读" → 继续阅读同主题文章
- 场景 2：搜索引擎引流的匿名用户 → 详情页停留 30s+ → 看到相关阅读 → 减少跳出
```

**差的例子**：
```
用户看完文章后会看到相关推荐。  ← 没说是哪种用户、什么场景
```

## 段落 4：验收标准

**目标**：定义"什么叫做完了"——验收时不再各说各话。

**原则**：
- 可观测：能看到具体结果
- 可复现：按步骤能重现
- 可度量：有数字或明确 checkbox

**好的例子**：
```
- [ ] 访问任意博客详情页 URL，页面底部出现"相关阅读"区块
- [ ] 相关阅读区块显示 4 篇文章卡片，每张卡片含标题 / 封面 / 发布日期
- [ ] 点击任一卡片，跳转到对应文章详情页，URL 含 `?from=related`
- [ ] Google Analytics 能看到 `related_click` 事件，参数包含 source_id 和 target_id
- [ ] 在无相似文章时，区块不展示（不要显示空状态）
- [ ] 移动端和桌面端布局都要通过 Pixel / iPhone 视口 screenshot 对比
```

**差的例子**：
```
- 相关阅读功能正常工作
- 用户体验良好    ← 主观、不可测
- 性能达标       ← 多少 ms 算达标？
```

## 段落 5：技术约束

**目标**：把技评前已知的技术边界明确下来，技评不用再猜。

**最小字段**：
- 涉及的服务 / 仓库（CMS / 前端 / 后端）
- 依赖的数据表 / API
- 性能约束（如有）
- 浏览器 / 终端约束（如有）

**好的例子**：
```
- 仓库：fe-tripo-studio（前端）+ CMS（新增字段 / API）
- 数据：`posts` collection 需要 `tags` 字段（已存在）
- 性能：相似度计算不能让详情页首屏超过 2s
- 兼容：Chrome 最近 3 个版本 + Safari 最近 2 个版本
```

## 段落 6：合规风险

**目标**：把所有已识别的合规 / 真实性风险显式列出来，让用户决策。

**好的例子**：
```
| 风险点 | 来源政策 | 违规后果 | 建议 |
|-------|---------|---------|------|
| 相似度分数展示 | Google 结构化数据指南 | 索引降权 | 只展示文章卡片，不显示分数 |
| 伪造阅读量（PRD 提到"显示 XX 人读过"） | 数据真实性 | 用户信任流失 | 建议移除，或只显示真实数据 |
```

**差的例子**：
```
无合规风险。  ← 没扫过 checklist，或者真的没风险？看不出来
```

## 段落 7：问题与风险

**目标**：把未澄清项和技术风险显式列出——进入下一步前必须解决或标记接受。

**好的例子**：
```
未澄清：
- 相关文章的排序策略：按相似度降序？还是相似度 + 时间混合？（需 Owner 确认）
- 移动端卡片布局是 2x2 还是横向滑动？（需设计稿）

技术风险：
- 相似度算法性能：如果 tags 是多对多，join 查询可能慢
- 冷启动问题：新文章没有相似文章时显示什么？（已在边界段约定"不展示"）
```

## 段落 8：评审结论

**目标**：一句话推动流程——是进入技评、回去澄清、还是打回产品？

**好的例子**：
```
- 进入技术评审：✅ 是
- 需要补充澄清：排序策略 / 移动端布局（已同步产品 Owner，预计 1 天内回复）
- 建议优先级：P1（不紧急但影响 SEO 核心指标）
```

**差的例子**：
```
可以开始开发。  ← 没有资源 / 风险结论
```
