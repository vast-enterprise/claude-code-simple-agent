# 风险识别清单：判断方法与处理建议

> SKILL.md 的展开篇。每条风险给出"如何判断是否命中"和"命中后怎么处理"。

## 合规与真实性（强制扫查）

### 1. 外部平台政策冲突

**怎么判断**：
- 需求是否要在 Google 搜索 / 社交媒体 / App Store 上展示内容？
- 需求是否要调用第三方 API（OpenAI / Stripe / Google Maps）？

**常见雷区**：
- 在 SEO 页面上展示**虚构的星级 / 评分**（违反 Google 结构化数据指南）
- 在 App Store 描述中宣传**未发布的功能**（违反审核规则）
- 在第三方 API 调用中**隐藏真实用户**（违反 TOS 中的 user-consent 条款）

**处理**：
- 在 PRD 合规风险段列出冲突点、引用具体政策 URL、给出"保留 / 移除 / 替代方案"三选项
- 不在评审阶段自行决策，让用户做取舍

### 2. 数据真实性风险

**怎么判断**：
- 需求是否要展示**用户无法验证**的数据？（"99% 用户推荐"、"10 万人在用"）
- 是否要**生成不存在**的数据支撑页面？（假评分、假评论、假用户）
- 是否使用**误导性统计**？（"增长 300%" 但基数是 10）

**红线**：
- 绝不接受"为了 SEO 效果生成假数据"的需求
- 发现就在 PRD 显式标注，建议"移除或用真实数据"
- 用户坚持要做时，把合规后果（索引降权 / 平台惩罚 / 法律风险）写进 PRD 让用户签字

### 3. 隐私数据

**怎么判断**：
- 是否涉及手机号 / 邮箱 / 地址 / 身份证 / 银行卡？
- 是否涉及用户的行为日志（页面访问 / 点击路径 / 停留时长）？
- 是否跨组织共享数据？

**处理**：
- 列出涉及的隐私字段，标注"需要脱敏 / 加密 / 授权"
- 跨组织共享需要用户明确同意（PRD 必须有同意流程设计）

### 4. 支付 / 财务数据

**怎么判断**：
- 是否涉及订单 / 支付 / 账单 / 发票？
- 是否涉及价格展示？折扣？优惠券？

**处理**：
- 支付需求必须有审计日志（所有写操作可追溯）
- 价格展示需要考虑精度（不能因为 JS 浮点数导致错价）

## 技术风险

### 5. 新依赖

**怎么判断**：
- 是否引入新的 npm / pip / cargo 包？
- 是否引入新的外部服务 / SaaS？
- 是否使用项目中未有的技术栈？

**处理**：
- 新依赖在技评阶段评估：包大小、维护状态、license、安全漏洞
- 能用现有依赖就别新增（每个新依赖都是未来的升级成本）

### 6. 跨仓库改动

**怎么判断**：
- 需求是否同时涉及前端 + 后端 + CMS？
- 是否需要多个 PR 配合发布？

**处理**：
- 在 PRD 技术约束段列出所有涉及的仓库
- 标注发布顺序（通常上游先于下游）
- 考虑分阶段发布的方案（先后端，一周后前端）

### 7. 破坏性变更

**怎么判断**：
- 是否修改已有 API 的返回格式？
- 是否删除 / 重命名数据库字段？
- 是否有线上数据需要迁移？

**处理**：
- 列出所有调用方（grep 代码 + 查埋点）
- 考虑向后兼容方案（新旧字段并存 N 周后再下掉）
- 数据迁移必须有回滚方案

### 8. 性能敏感点

**怎么判断**：
- 是否有列表查询？（可能 N+1）
- 是否有大表扫描？（> 100 万行）
- 是否是高频调用的接口？（首页 / 详情页）

**处理**：
- 在 PRD 技术约束段标注性能指标（P50 < 100ms，P99 < 500ms）
- 要求技评阶段提供压测方案

## 资源与协作

### 9. 跨团队配合

**怎么判断**：
- 需要设计师出图？
- 需要运维改配置 / 开端口？
- 需要数据团队准备数据 / 埋点？

**处理**：
- 在 PRD 列出依赖方和交付物 deadline
- 提前同步对方（别等开始做了才问）

### 10. 数据 ready 度

**怎么判断**：
- 需求依赖的数据**现在存在吗**？
- 格式一致吗？有没有脏数据？
- 需要先跑一次迁移 / 回填吗？

**处理**：
- 技评前先抽样看数据（别靠"应该有"猜）
- 脏数据率超过 5% 时，数据清洗单独立项

### 11. 时间冲突

**怎么判断**：
- 目标上线日期在发车窗口期内吗？
- 上线日期是否撞上封板期 / 合规审查期 / 节假日？
- 上线日期与其他大需求是否有冲突？

**处理**：
- 在 PRD 列出目标上线日 + 可用发车窗口
- 冲突时给出"推迟 / 降级 / 切分"选项

## 验收风险

### 12. 验收标准不可测

**怎么判断**：
- 验收标准里有"体验好" / "美观" / "流畅"这类主观词吗？
- 验收标准是否可观测？能否自动化测试？

**处理**：
- 把主观词改成可观测标准（"流畅" → "页面加载 < 2s"）
- 全部改不了的，用截图对比 / UAT 验收

### 13. 无灰度 / 回滚方案

**怎么判断**：
- 功能上线后如果不符合预期，能快速关掉吗？
- 影响的是全量用户还是可以灰度？

**处理**：
- 关键功能必须有 feature flag（可即时关闭）
- 大需求必须分 10% / 50% / 100% 灰度

### 14. 数据回滚能力

**怎么判断**：
- 新功能会写入哪些数据？
- 这些数据能撤销吗？用户看过后撤销有影响吗？

**处理**：
- 涉及"已发布内容 / 已发送消息"的写操作需要明确"是否可撤销"
- 不可撤销的操作必须有二次确认

## 使用方法

评审阶段按清单过一遍，每条回答"命中 / 不命中 / 不确定"：

- **命中**：在 PRD 合规风险段或问题与风险段列出
- **不命中**：跳过
- **不确定**：用 AskUserQuestion 问用户或同步相关岗位

不写"无风险"——要么是扫过了都不命中（一条条说明），要么是没扫（需要扫）。
