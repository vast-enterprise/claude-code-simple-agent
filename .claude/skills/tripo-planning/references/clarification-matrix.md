# 12 澄清维度：问题模板与反模式

> SKILL.md 的展开篇。每个维度给出具体的 AskUserQuestion 问题模板和常见脑补陷阱。

## 1. 用户

**要问清楚**：
- 这个功能给谁用？B 端（内部工具 / 运营后台）？C 端（注册用户 / 匿名访客）？
- 同一时间大概多少人用？（高并发设计 vs 单点低频）
- 用户的技术熟练度？（影响交互复杂度）

**反模式**：
- 脑补"肯定是给所有人用的"——可能只给特定角色
- 假设"用户会知道怎么用"——可能需要引导 / tutorial

## 2. 场景

**要问清楚**：
- 什么业务场景会触发这个功能？
- 是日常高频使用（每天多次）还是偶发场景（一周一次）？
- 是主路径还是边缘路径？

**反模式**：
- 把"运营偶尔用"的工具当成"高频" —— 过度设计
- 把"用户每天用"的功能当成"辅助" —— 体验 / 性能不达标

## 3. 目标

**要问清楚**：
- 做成之后衡量成功的指标是什么？（转化率 / 留存 / 效率提升 / 收入）
- 指标怎么度量？从哪里看？
- 多少算成功？

**反模式**：
- "提升用户体验"—— 不可度量，等于没目标
- 没问目标直接进评审 —— 后续功能取舍无依据

## 4. 核心功能

**要问清楚**：
- 必须要做的最小功能集是什么？
- 哪些是"nice to have"？

**反模式**：
- 把 PRD 里列的所有点当成"都必做"——实际可能有优先级差异
- 遗漏"隐含前提"（如"需要先有用户登录"）

## 5. 边界

**要问清楚**：
- 哪些看似相关但这次**不做**？
- 历史遗留问题要不要顺带修？
- 其他平台 / 端 / 入口要不要同步？

**反模式**：
- 默认"都要做"—— 范围无限蔓延
- 默认"都不做"—— 漏掉用户期望的隐含项

## 6. 验收标准

**要问清楚**：
- 怎样叫"做完了"？
- 验收时怎么测？谁测？
- 有没有数据 / 性能门槛？

**判断验收标准是否合格**：
- 可观测（能看到结果）
- 可复现（别人按步骤能重现）
- 不含主观词汇（"流畅" / "美观" / "体验好"）

**反模式**：
- "页面能正常显示"——什么叫正常？
- "性能不错"——不错是多少 ms？

## 7. 依赖

**要问清楚**：
- 依赖哪些内部服务 / 数据库 / 表？
- 依赖哪些第三方 API？有没有 SLA？
- 依赖的数据**现在存在吗**？还是需要先建？

**反模式**：
- 假设"数据都有"——结果需要先跑迁移
- 假设"API 已经 ready"——结果是别的团队还没开始做

## 8. 数据

**要问清楚**：
- 要读哪些表？
- 要写哪些表 / 新增字段？
- 数据格式约束？（字段长度 / 枚举值）
- 历史数据需要迁移 / 回填吗？

**反模式**：
- 忽略"历史数据处理" —— 上线那天发现老数据格式不兼容
- 忽略"字段上限" —— VARCHAR(255) 撑不住

## 9. 权限

**要问清楚**：
- 谁能看？登录还是不登录？
- 谁能改？需要什么角色？
- 跨组织 / 跨租户有没有隔离要求？

**反模式**：
- 默认"内部工具不用权限" —— 真的有人误操作
- 权限设计过细 —— 维护成本飙升

## 10. 异常

**要问清楚**：
- 请求失败用户看到什么？
- 数据不存在展示什么？
- 有没有兜底 / 降级方案？

**反模式**：
- "正常情况下不会出错" —— 永远会出错
- 异常态展示"系统错误" —— 用户不知道怎么办

## 11. 合规与真实性（强制）

**要问清楚**：
- 是否涉及外部平台政策？（Google 结构化数据 / App Store / 第三方 TOS）
- 是否有**虚假数据**嫌疑？（伪造评分、虚构用户数、捏造统计指标）
- 是否涉及隐私 / 支付数据？

**数据真实性红线**：
- 绝不生成不存在的用户数据 / 评分 / 访问量
- 绝不使用误导性的统计描述（"99% 用户" 但没有数据支持）
- 如果需求方要求，先在 PRD 合规风险段标注，让用户决策

**反模式**：
- 把"增加可信度"当成可以造数据 —— 这是合规红线
- 忽略平台政策 —— 可能导致索引降权 / 审核驳回

## 12. 时间压力

**要问清楚**：
- 什么时候要上？（具体日期 / Sprint）
- 紧急程度？（hotfix / 跟车 / 下个 Sprint）
- 有没有外部 deadline？（合规日期 / 营销节点）

**反模式**：
- 不问时间直接方案 —— 方案过重 / 过轻
- 时间压力不告诉开发 —— 开发自然按常规节奏走

## AskUserQuestion 的使用节奏

不要一次抛 12 个问题（会被秒拒或答非所问）。建议：

1. **第一轮**：先识别 2-3 个最关键的未知维度（通常是用户 / 目标 / 边界）
2. **第二轮**：基于第一轮答案，再问 2-3 个衍生问题（通常是依赖 / 数据 / 验收）
3. **第三轮**：补齐异常 / 权限 / 时间等执行细节

每轮 AskUserQuestion 的 `question` 字段写得具体、带示例，比开放式问题更能拿到有用答案。
