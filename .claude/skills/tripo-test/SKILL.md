# 集成测试方法论

## 先对齐再测试

测试前必须确认两件事对齐：用户期望验证什么（预期结果）、我实际要测什么（测试操作）。

如果有测试计划文档，先读取计划，确认本轮场景在计划中。没有计划的，先输出计划再执行。

## 什么是好的测试计划

测试计划不是"列几个要测的东西"。好的计划对每个场景回答三个问题：

1. **验证什么** — 对照下方分类表确定测试类型，不能全用 API 测试糊弄
2. **用什么工具、留什么证据** — 分类表的工具和证据列就是答案
3. **测试数据怎么准备** — 按"测试数据"一节的要求，不是随便填

缺少任何一项的计划 = 没有计划。

## 选对工具

根据验证对象选测试类型。不能图省事用错工具——curl 能拿到 HTML 不代表验证了 UI。

| 验证对象 | 测试类型 | 工具 | 证据形式 |
|---------|---------|------|---------|
| HTTP 接口响应数据 | API 测试 | curl / httpie | 请求命令 + 响应 body |
| 页面元素存在性、样式 | UI 渲染测试 | playwright snapshot + screenshot | snapshot 输出 + 截图 |
| 操作后 UI 状态变化 | UI 交互测试 | playwright click/fill + screenshot | 操作前后截图对比 |
| meta/OG/结构化数据 | SEO 测试 | playwright eval | eval 返回值 |
| 端到端数据流 | 跨仓库联调 | 组合以上 | 分层验证：API → SSR → 交互 → SEO，每层一份证据 |

**关键判断**：SEO meta 标签可能在客户端 JS 执行后才注入，curl 只拿到 SSR HTML——必须用浏览器验证。

## 证据标准

**铁律：证据先贴，结论后出。** 测试要有对比——预期值 vs 实际值、操作前 vs 操作后、正常 case vs 错误 case。

| 有效证据 | 无效证据 |
|---------|---------|
| curl 请求 + 完整响应 | "我验证过了" |
| playwright screenshot 截图 | "页面加载正常" |
| playwright snapshot DOM 输出 | "功能工作" |
| console 日志截取 | "没有报错" |
| 测试命令 + 完整输出 | 对过程的文字描述 |

## 测试环境纪律

**本地优先，staging 是例外不是默认。** 测试环境按下面的优先级选，跳级必须贴理由。

| 场景 | 默认环境 | 例外 |
|------|----------|------|
| 开发中 / PR 验证 / bugfix 回归 | **本地 dev server** | 跳到 staging 必须贴：本地启动失败的完整日志 + 至少 2 次排障尝试 + 用户授权 |
| 上线前最终验收 | staging | — |
| 生产验证（curl 线上 URL / 看 pod 日志） | production | 仅 release / 发车场景 |

**铁律：没试过本地启动就建议上 staging = 用环境差异当借口，不是测试。** 遇到本地起不来，先走三步排障：

1. 读目标仓库的 `README.md` / `llmdoc/guides/` 找启动命令
2. 看报错日志 → 检查依赖（`pnpm install`）、端口占用、环境变量（`.env.local`）
3. 排障仍失败 → AskUserQuestion 说明具体报错 + 问是否授权上 staging

未完成这三步而直接推 staging，不算"测试 DEFERRED"，算"测试缺席"。

## 测试数据

### 数据形状

按组件 props / API 参数的完整要求准备，不是随便填一个：

- 必填字段全部赋值
- 至少一个可选字段用非默认值（验证非默认路径）
- 至少一个边界值（空字符串、超长文本、特殊字符）
- 不只测 happy path，异常场景也要覆盖

### 数据缺失时的构造义务

**"数据库里没这条记录"不是降级理由，是测试准备的一部分。** 遇到测试数据不存在，按下面的顺序处置，跳级必须贴理由：

1. **用现有种子数据**：查 CMS 种子 / 数据库 seed / fixture 文件是否已有满足条件的记录
2. **主动构造**：通过 API（CMS Payload REST / 业务 API）或 CMS UI 构造一条最小可用数据；构造脚本 / curl 命令留在报告里
3. **请 developer 补 seed**：若构造涉及复杂权限 / 跨表 / 写死常量，向 developer 申请补 seed 或 mock 数据
4. **最后才 DEFERRED**：以上三步都失败，贴三次尝试证据 + 具体障碍 + 用户授权，才允许标 DEFERRED

**滥用"依赖外部服务且不可 mock"把能构造的数据包装成降级 = 把测试义务转嫁给下次的人。** 区分"外部依赖不可 mock"（如三方支付沙箱挂了）和"数据不存在"（我方系统里可以造）——后者必须造。

## 工具纪律

1. **用户指定了工具 → 必须用该工具**，遇到障碍先排障，不私自替换
2. **排障失败 → 先向用户说明**，获得许可后才能用替代方案
3. **替代方案必须标注**：在报告中写明 `⚠️ 工具替代: <原因>`

## FAIL 分流

测试场景 FAIL 时：

- 失败原因明确（返回值不对、缺字段、样式错位）→ 修复后重测
- 失败原因不明 → 加载 tripo-diagnose 做根因分析

## 降级条件

DEFERRED 不是避让通道，是**排障/构造都失败后**的最后记账。每类 DEFERRED 必须先完成前置义务并贴证据，否则不算降级、算缺席。

| DEFERRED 类型 | 前置义务（缺一不可） | 证据形式 |
|---------------|-----------------------|----------|
| 服务无法启动 | 已尝试"测试环境纪律"三步排障 | 启动命令 + 完整报错日志 + 排障记录 |
| 数据不存在且不可构造 | 已走完"数据缺失时的构造义务"四步 | 三次构造尝试的命令 + 失败输出 + 具体障碍 |
| 依赖外部服务且不可 mock | 明确是**我方系统外**的依赖（三方 API、沙箱环境） | 依赖名 + 不可 mock 的具体原因（不是"懒得造数据"） |
| 工具替代 | 用户已授权 | 替代原因 + 授权记录 |

**所有 DEFERRED 都必须附补测计划：什么条件满足后回来补测、由谁触发。**
