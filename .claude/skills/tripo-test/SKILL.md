# 集成测试方法论

## 先对齐再测试

测试前必须确认两件事对齐：用户期望验证什么（预期结果）、我实际要测什么（测试操作）。

如果有测试计划文档，先读取计划，确认本轮场景在计划中。没有计划的，先输出计划再执行。

## 什么是好的测试计划

测试计划不是"列几个要测的东西"。好的计划对每个场景回答三个问题：

1. **验证什么** — 对照下方分类表确定测试类型，不能全用 API 测试糊弄
2. **用什么工具、留什么证据** — 分类表的工具和证据列就是答案
3. **测试数据怎么准备** — 按"测试数据"一节的要求，不是随便填

缺少任何一项的计划 = 没有计划。

## 选对工具

根据验证对象选测试类型。不能图省事用错工具——curl 能拿到 HTML 不代表验证了 UI。

| 验证对象 | 测试类型 | 工具 | 证据形式 |
|---------|---------|------|---------|
| HTTP 接口响应数据 | API 测试 | curl / httpie | 请求命令 + 响应 body |
| 页面元素存在性、样式 | UI 渲染测试 | playwright snapshot + screenshot | snapshot 输出 + 截图 |
| 操作后 UI 状态变化 | UI 交互测试 | playwright click/fill + screenshot | 操作前后截图对比 |
| meta/OG/结构化数据 | SEO 测试 | playwright eval | eval 返回值 |
| 端到端数据流 | 跨仓库联调 | 组合以上 | 分层验证：API → SSR → 交互 → SEO，每层一份证据 |

**关键判断**：SEO meta 标签可能在客户端 JS 执行后才注入，curl 只拿到 SSR HTML——必须用浏览器验证。

## 证据标准

**铁律：证据先贴，结论后出。** 测试要有对比——预期值 vs 实际值、操作前 vs 操作后、正常 case vs 错误 case。

| 有效证据 | 无效证据 |
|---------|---------|
| curl 请求 + 完整响应 | "我验证过了" |
| playwright screenshot 截图 | "页面加载正常" |
| playwright snapshot DOM 输出 | "功能工作" |
| console 日志截取 | "没有报错" |
| 测试命令 + 完整输出 | 对过程的文字描述 |

## 测试数据

按组件 props / API 参数的完整要求准备，不是随便填一个：

- 必填字段全部赋值
- 至少一个可选字段用非默认值（验证非默认路径）
- 至少一个边界值（空字符串、超长文本、特殊字符）
- 不只测 happy path，异常场景也要覆盖

## 工具纪律

1. **用户指定了工具 → 必须用该工具**，遇到障碍先排障，不私自替换
2. **排障失败 → 先向用户说明**，获得许可后才能用替代方案
3. **替代方案必须标注**：在报告中写明 `⚠️ 工具替代: <原因>`

## FAIL 分流

测试场景 FAIL 时：

- 失败原因明确（返回值不对、缺字段、样式错位）→ 修复后重测
- 失败原因不明 → 加载 tripo-diagnose 做根因分析

## 降级条件

仅以下情况允许标记 `⚠️ DEFERRED`（附原因和补测计划）：

- 服务无法启动（贴启动日志）
- 依赖外部服务且不可 mock（说明具体依赖）
- 工具替代已获用户许可（标注替代原因）
