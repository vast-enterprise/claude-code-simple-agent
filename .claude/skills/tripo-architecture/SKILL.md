# 技术方案设计方法论

## 总则：方案不是第一版，是被对比出来的那一版

方案设计不是"想到一个思路写下来"，是**把至少 2 个候选方案放在同一张表上对比，按明确维度权衡，选出理由最站得住的那个**。一个需求只有一份方案，大概率是"第一想法直接落文字"——这类方案通常在实施 / 验收阶段被证伪，返工成本远高于多方案对比的成本。

对比先于推荐，推荐先于详设，详设先于回退——这个顺序不能反。

## 技术方案文档结构（technical-solution.md 内容标准）

方案文档是 PRD 的下游产物，结构固定：

```markdown
# <需求名> 技术方案

## 需求概述
- 引用 review.md 的核心功能 + 验收标准（一句话）
- PRD 中识别的关键风险（合规 / 技术 / 资源）
- 目标上线时间

## 涉及范围

### 仓库
| 仓库 | 是否改动 | 影响模块 |
|------|---------|---------|
| fe-tripo-studio | ✅ | pages/blog/[slug] |
| CMS | ✅ | posts collection + REST API |
| 其他 | ❌ | - |

### 跨仓库契约（如有）
- API: `GET /api/posts/:id/related` —— 返回格式、字段语义、错误码
- 数据流：上游写入 → 下游读取的字段映射

## 方案对比

至少 2 个候选方案，按固定维度对比：

| 维度 | 方案 A：<一句话描述> | 方案 B：<一句话描述> |
|------|---------------------|---------------------|
| 实现思路 | ... | ... |
| 工作量（P50 / P90） | 3d / 5d | 5d / 8d |
| 性能预期 | P99 < 200ms | P99 < 500ms |
| 可维护性 | 依赖现有 Elasticsearch | 新引入向量数据库 |
| 可回退性 | feature flag 可即时关 | 数据迁移后不可回退 |
| 跨仓库影响 | 仅前端 | 前端 + CMS + 新服务 |
| 破坏性变更 | 无 | 改 API 返回格式 |
| 合规与安全 | 无 | 涉及用户行为数据 |
| 成熟度 | 团队熟悉 | 需学习 |

**推荐：方案 A**
**推荐理由**：工作量低、可回退、团队熟悉；方案 B 的性能优势在当前场景不关键（需求量级 QPS < 10）。

## 详细设计（基于推荐方案）

### 涉及模块
- 前端：`src/pages/blog/[slug]/RelatedPosts.tsx`
- 后端：`CMS/src/collections/posts/hooks.ts`
- 数据库：`posts` collection 新增 `relatedCache` 字段

### 接口设计
```
GET /api/posts/:id/related?limit=4
Response: { items: [{id, title, coverUrl, publishedAt}], total: 4 }
Error: 404 (post not found) / 503 (cache miss + compute timeout)
```

### 数据模型变更
- 新增字段：`posts.relatedCache: JSON | null`
- 写入时机：文章发布时计算并缓存；tag 变更时失效
- 迁移脚本：`migrations/2026-04-20-add-related-cache.ts`

### 关键流程
- 序列图 / 伪代码（如有复杂交互）

## 工作量评估

| 任务 | P50 | P90 | 依赖 |
|------|-----|-----|------|
| CMS 新增字段 + 迁移 | 0.5d | 1d | - |
| 相似度算法实现 | 1d | 2d | 字段就绪 |
| 前端组件 + 联调 | 1d | 1.5d | API 就绪 |
| 测试 + bugfix | 0.5d | 1d | 全部完成 |
| **关键路径（P90）** | - | **5.5d** | - |

## 风险评估

| 风险 | 触发概率 | 影响 | 应对 |
|------|---------|------|------|
| 相似度算法冷启动慢 | 中 | 首次发布文章时首屏慢 | 异步计算，首次展示 fallback |
| tag 字段脏数据 | 低 | 相似度算结果偏差 | 上线前跑一次数据质量扫描 |
| API 契约变更影响其他端 | 低 | 移动端报错 | 新增 endpoint，不改老的 |

## 回退方案

- **feature flag**：`enable_related_posts`，默认 off
- **数据回退**：`relatedCache` 字段可为 null，不影响现有查询
- **回退 SLA**：生产发现问题，10 分钟内关 flag，24 小时内决定是否删字段

## 测试计划

（测试类型选择、覆盖范围——具体执行交给 tester）
- 单元测试：相似度算法
- 集成测试：API endpoint
- E2E：详情页看到相关阅读模块

## 安全与合规 checklist

- [ ] 是否涉及用户隐私数据？（本需求：否）
- [ ] 是否涉及支付 / 财务？（本需求：否）
- [ ] 是否有权限控制？（本需求：相关阅读仅展示已发布文章，无额外权限）
- [ ] 是否有合规红线？（本需求：PRD 已标注，无）
- [ ] 外部平台政策？（本需求：不涉及）
- [ ] 输入校验？（本需求：limit 参数需要上限校验）
- [ ] SQL / NoSQL 注入？（本需求：使用 ORM，风险低）
- [ ] XSS？（本需求：渲染文章标题需要转义）
```

> 详见 [references/solution-template.md](references/solution-template.md) —— 每段落的填充原则和常见问题。

## 方案对比的固定维度

无论什么需求，候选方案对比必须覆盖这 9 个维度。缺一个维度就是把这个维度的权衡推给了实施阶段。

| 维度 | 问"什么" |
|------|---------|
| **实现思路** | 核心技术路径是什么？一句话描述 |
| **工作量（P50 / P90）** | 顺利情况 / 踩坑情况的工时估算 |
| **性能预期** | P99 延迟、内存占用、扫描行数、QPS 上限 |
| **可维护性** | 依赖既有技术栈还是新引入？团队熟悉度？ |
| **可回退性** | feature flag？数据能撤销？回退成本？ |
| **跨仓库影响** | 动了哪些仓库？需要协调几个 PR？ |
| **破坏性变更** | 改了哪些既有 API / schema？对调用方影响？ |
| **合规与安全** | 涉及隐私数据 / 外部平台政策 / 数据真实性？ |
| **成熟度** | 方案在团队 / 行业中的成熟度？有无成功案例？ |

维度缺失 = 方案对比不完整。评审时应该被打回补齐。

> 详见 [references/comparison-matrix.md](references/comparison-matrix.md) —— 每个维度的量化方法。

## 跨仓库影响分析法

跨仓库改动是事故高发区。方案设计阶段必须显式扫查，不留给开发阶段发现。

### 三步扫查

**第一步：仓库清单**
- 按需求的"核心功能"倒推涉及哪些仓库
- 参考 `tripo-repos` skill 里的仓库注册表
- 漏一个仓库 = 开发时才发现要动 N 个仓库

**第二步：契约设计**
- 两个仓库之间的接口 / 数据格式 / 消息协议叫"契约"
- 契约必须在方案阶段明确：字段名、类型、错误码、版本策略
- 不能"先上游改，下游再对齐"——这是返工的经典模式

**第三步：发布顺序**
- 上游变更 → 下游适配 → 上游清理老接口
- 不是一个 PR 跨仓库，而是多个 PR 有顺序地发布
- 方案文档里写清发布顺序和时间窗口

### 常见反模式

| 反模式 | 问题 | 纠正 |
|-------|------|------|
| 一个 PR 跨仓库 | 无法独立回退 | 拆成多个 PR，按依赖顺序合入 |
| 上游先改再说 | 下游开发时发现不兼容 | 方案阶段先定契约，两边并行开发 |
| 直接改老 API 格式 | 老调用方全部炸 | 新增 endpoint 或加版本号 |
| 数据格式无版本 | 未来改时无法兼容 | 所有契约自带 schema version |

## 技术选型矩阵

选型不是拍脑袋，是按权重打分。每个技术选型决策至少按 4 维度打分：

| 维度 | 权重 | 评估方式 |
|------|------|---------|
| **成熟度** | 30% | GitHub star / 社区活跃度 / 生产使用案例 |
| **契合度** | 30% | 跟现有技术栈匹配度 / 团队熟悉度 / 迁移成本 |
| **学习成本** | 20% | 新人上手时间 / 文档质量 / 调试难度 |
| **风险** | 20% | license / 安全漏洞历史 / 维护者数量 |

每个候选方案按 1-5 分打分，加权后选最高分的。不打分直接"凭感觉选 X"，是在把选型风险推给未来的自己。

> 详见 [references/selection-matrix.md](references/selection-matrix.md) —— 常见技术栈的参考分值。

## 安全与合规 checklist（强制）

方案阶段必过的清单。未过 checklist 的方案不能进入开发。

### 安全
- [ ] 所有用户输入都在系统边界做了校验？
- [ ] 有无 SQL / NoSQL / 命令注入风险？
- [ ] 渲染用户内容时做了转义（XSS）？
- [ ] CSRF / 跨站 token 设计？
- [ ] 鉴权与授权分开处理？
- [ ] 错误信息不泄露内部实现 / 堆栈？
- [ ] 速率限制（rate limiting）？
- [ ] 敏感数据加密（存储 / 传输）？

### 合规
- [ ] 合规风险 PRD 已标注？方案如何实现？
- [ ] 外部平台政策（Google / App Store / 第三方 TOS）遵守？
- [ ] 数据真实性：方案里没有"生成假数据"的环节？
- [ ] 隐私数据脱敏 / 加密 / 授权流程？
- [ ] 审计日志（写操作可追溯）？

### 可观测
- [ ] 关键路径有日志（error / warn 级别）？
- [ ] 关键指标有埋点（成功率 / 延迟）？
- [ ] 告警阈值设定？

## 我要不要动代码？

**不动。** `tripo-architecture` 是方法论 skill，服务于 architect agent。architect 的产出是方案文档，不是代码。写代码交给 developer agent（加载 `tripo-dev`）。

想"先写点 demo 验证一下"——停。demo 如果是方案验证必需的，单独开 POC 任务，由 developer 承接；不要在方案阶段顺手写代码，会让"方案"和"实现"混成一团。
