# 技术选型矩阵：加权打分法

> SKILL.md 的展开篇。给出每个选型场景的打分参考和决策示例。

## 打分方法

四个维度，每个 1-5 分，按权重加权：

| 维度 | 权重 |
|------|------|
| 成熟度 | 30% |
| 契合度 | 30% |
| 学习成本 | 20% |
| 风险 | 20% |

加权总分 = 成熟度 × 0.3 + 契合度 × 0.3 + 学习成本 × 0.2 + 风险 × 0.2

## 各维度打分参考

### 成熟度（30%）

| 分数 | 标准 |
|------|------|
| 5 | 行业标准（React / PostgreSQL 级别），有大量生产案例 |
| 4 | 主流选择，社区活跃，star 10k+ |
| 3 | 有一定社区，star 1k-10k，少量生产案例 |
| 2 | 小众，star < 1k，主要是个人项目 |
| 1 | 新项目或 beta，文档不全 |

### 契合度（30%）

| 分数 | 标准 |
|------|------|
| 5 | 跟现有技术栈无缝（同语言 / 同框架生态） |
| 4 | 同生态，需少量适配 |
| 3 | 跨生态但有成熟集成方案 |
| 2 | 跨生态，需要写 adapter |
| 1 | 技术栈冲突，需要重大改造 |

### 学习成本（20%）

分数越高表示学习成本越低。

| 分数 | 标准 |
|------|------|
| 5 | 团队已用过，无新人上手成本 |
| 4 | 文档优秀，新人 1 天上手 |
| 3 | 文档 OK，新人 3 天上手 |
| 2 | 文档一般，需要查源码，新人 1 周上手 |
| 1 | 文档差，调试难，需要专人投入 |

### 风险（20%）

分数越高表示风险越低。

| 分数 | 标准 |
|------|------|
| 5 | 商业 license，大厂维护，无已知严重漏洞 |
| 4 | 主流 OSS license（MIT/Apache），活跃维护 |
| 3 | OSS license，维护者 > 3 人 |
| 2 | 维护者 1-2 人，漏洞响应慢 |
| 1 | 维护不活跃，有未修复漏洞，或 license 有争议 |

## 示例：缓存方案选型

场景：博客详情页的相关阅读需要缓存。

### 候选

- A：不缓存（直接查库）
- B：内存缓存（Node.js 内存 Map）
- C：Redis
- D：预计算 + DB 字段

### 打分

| 维度 | A | B | C | D |
|------|---|---|---|---|
| 成熟度 | 5 | 5 | 5 | 5 |
| 契合度 | 5 | 5 | 3（需新建） | 5 |
| 学习成本 | 5 | 5 | 3 | 5 |
| 风险 | 5 | 4（无跨实例） | 3（多一个依赖） | 5 |
| **加权** | **5.0** | **4.8** | **3.6** | **5.0** |

A 和 D 都是 5.0，但 A 性能不达标（PRD 要求 P99 < 200ms，直接查库 > 500ms）——排除。

选 D：预计算 + DB 字段。

### 权衡记录

"放弃 Redis 的极致性能，换取更低的运维复杂度和零新增依赖。"

## 反模式

| 反模式 | 问题 | 纠正 |
|-------|------|------|
| 直接选最熟悉的 | 可能错过更合适的方案 | 至少 2 个候选打分 |
| 只看成熟度不看契合度 | 行业标准但不适合团队 | 契合度权重一样高 |
| 忽略学习成本 | 新人上手慢拖进度 | 纳入权重 |
| 不看 license | 商用合规风险 | 风险维度覆盖 license |
| "项目组 leader 说用 X" | 缺少依据，无法 handover | 仍需走打分流程，把 leader 的判断作为契合度输入 |

## 什么时候不需要打分矩阵

- 只有一个候选（无替代品）
- 团队有强制规范（比如公司统一 MySQL，就不用对比 PG）
- 改动极小（只是加个配置）

这些情况在方案文档里显式标注"无需选型：<原因>"，不是跳过这段。
