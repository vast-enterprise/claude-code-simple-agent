# 方案对比矩阵：9 维度量化方法

> SKILL.md 的展开篇。每个维度给出具体的评估方式和常见陷阱。

## 1. 实现思路

**怎么写**：
- 一句话概括核心技术路径
- 点出关键组件和数据流

**好**："预计算 + 缓存字段：发布时算好存库，读时直接返回"
**差**："用算法算相关性"（没路径没数据流）

## 2. 工作量（P50 / P90）

**P50 标准**：代码熟悉度中等、没有阻塞依赖、一次通过 review
**P90 标准**：踩 2-3 个常见坑、需要返工 1 轮、依赖方延期 0.5d

**量化方法**：
- 0.5d 以下的粒度不单独列（合并到相邻任务）
- P90 不应小于 P50 × 1.5（小于说明没真考虑风险）
- 总和看关键路径（依赖链上的 P90 相加）

**常见陷阱**：
- P50 = P90：等于没做风险评估
- 忽略测试 / 迁移 / 文档：只估业务代码时长

## 3. 性能预期

**要量化**：
- 延迟：P50 / P99
- 吞吐：QPS 上限
- 资源：内存 / CPU / 存储增量

**数据来源**：
- 参考现有同类功能的监控数据
- 没数据的做 POC 压测
- 压测也不现实的标注"上线后观测"

**常见陷阱**：
- "性能 OK"：多少 ms 算 OK？
- 只估正常情况，不考虑冷启动 / 缓存未命中

## 4. 可维护性

**评估维度**：
- 依赖既有技术栈（+）vs 新引入（-）
- 团队熟悉度（熟悉 + / 陌生 -）
- 代码复杂度（简单 + / 复杂 -）
- 依赖文档质量（好 + / 差 -）

**量化**：1-5 分
- 5：团队熟，现有栈，代码简单
- 3：团队一半熟悉
- 1：全新技术，文档差

## 5. 可回退性

**评估维度**：
- 有 feature flag（+）
- 数据可撤销（+）vs 数据迁移不可逆（-）
- 代码回滚即生效（+）vs 需要 DB 迁移配合（-）

**量化**：1-5 分
- 5：flag 可即时关，无数据变更
- 3：代码回滚 10 分钟内生效
- 1：数据迁移后无法回退

**红线**：可回退性 = 1 的方案需要 PRD 显式授权

## 6. 跨仓库影响

**评估维度**：
- 涉及仓库数量
- 是否需要协调发布顺序
- 是否改契约（API 签名 / 数据格式）

**量化**：涉及仓库数 + 契约变更数
- 1 个仓库 + 无契约：影响小
- 2+ 仓库 + 改契约：影响大，需要分阶段发布

## 7. 破坏性变更

**识别**：
- 改已有 API 的返回格式？
- 删 / 改已有数据库字段？
- 改已有枚举值？
- 改已有行为（不是新增行为）？

**量化**：
- 无破坏性：优先选择
- 有破坏性：必须有兼容策略（新增 endpoint / schema version）

## 8. 合规与安全

**评估维度**：
- 是否涉及隐私 / 支付 / 合规数据？
- 是否有外部平台政策冲突？
- 是否引入新的攻击面？

**量化**：
- 无风险：跳过
- 有风险：必须有缓解措施 + checklist 全过

## 9. 成熟度

**评估维度**：
- 技术本身的社区成熟度（star 数 / 生产案例）
- 在本团队的使用历史（用过 + / 第一次 -）
- 行业内的成功案例

**量化**：1-5 分
- 5：团队主力技术，生产用过 > 1 年
- 3：团队用过但非主力，生产用过 < 6 个月
- 1：第一次用，无团队经验

## 推荐理由的写法

推荐某个方案时，理由必须基于对比矩阵的**具体维度差异**，不是"我觉得 A 更好"。

**好**：
```
推荐方案 A，因为：
- 工作量：A（P90 5.5d）< B（P90 8d），早 2.5d 上线
- 可回退性：A（null 字段即可回退）> B（依赖 Redis，回退需保留组件）
- 成熟度：A 使用现有 MySQL，B 新引入 Redis（团队用过但非主力）
- 性能：B 更快，但本需求 QPS < 10，A 的 P99 200ms 已足够

权衡：放弃 B 的性能优势，换取更低的工作量和更强的可回退性。
```

**差**：
```
推荐方案 A，比较简单。
```
