# 技术方案模板：各段落填充原则

> SKILL.md 的展开篇。给每个段落的"好 vs 差"对比。

## 段落 1：需求概述

**目标**：让评审人 30 秒内知道这个方案要解决什么。

**最小字段**：
- 引用 review.md 的核心功能 + 验收标准（一句话）
- PRD 已识别的关键风险
- 目标上线时间

**好的例子**：
```
需求来源：review.md（recXYZ123）
核心功能：博客详情页底部新增相关阅读模块，展示 4 篇相似文章
验收关键点：相似度基于 tag 重合度 + 发布时间衰减；点击埋点完整
PRD 关键风险：无合规风险；依赖 posts.tags 字段数据质量
目标上线：2026-05-10 前
```

**差的例子**：
```
做一个相关阅读功能。  ← 评审人要跳回 PRD 才能理解
```

## 段落 2：涉及范围

**目标**：把所有受影响的仓库、模块、契约显式列出，漏一个就是返工源头。

**好的例子**：
```
### 仓库
| 仓库 | 是否改动 | 影响模块 |
|------|---------|---------|
| fe-tripo-studio | ✅ | pages/blog/[slug]/RelatedPosts.tsx（新增） |
| CMS | ✅ | posts collection hooks + related endpoint |

### 跨仓库契约
- GET /api/posts/:id/related —— 前端调用，CMS 提供
- Response: { items: RelatedPost[], total: number }
- 错误：404（文章不存在） / 503（计算超时）
```

**差的例子**：
```
前后端都要改。  ← 没有具体模块，没有契约
```

## 段落 3：方案对比

**目标**：让评审人看得出"为什么不选另一个"。

**关键原则**：
- 至少 2 个候选方案（一个方案 = 没对比）
- 9 个维度都填（缺维度 = 对比不完整）
- 明确推荐哪个 + 为什么

**好的例子**：
```
### 方案 A：预计算 + 缓存字段
实现思路：文章发布时同步计算相似度 top 4，写入 relatedCache 字段
优点：读取快（无计算）、可回退（字段可为 null）
缺点：tag 变更时需要失效；存储略增

### 方案 B：实时查询 + Redis 缓存
实现思路：每次请求现查 tag 匹配 + top 4，Redis 缓 1h
优点：数据新鲜
缺点：Redis 依赖；首次无缓存慢

（对比表省略）

**推荐：方案 A**
**理由**：
- 当前需求 QPS < 10，方案 A 性能已足够
- 方案 A 不引入新依赖（Redis 需新建），契合度更高
- 可回退性：方案 A 可通过 null 字段回退，方案 B 依赖外部组件
```

**差的例子**：
```
我们用 Redis 缓存。  ← 没对比，没理由
```

## 段落 4：详细设计

**目标**：基于推荐方案，把实施细节写到 developer 能照着写的程度。

**最小要素**：
- 涉及的具体文件路径（而不是"前端改一下"）
- 接口 signature（URL / method / 请求体 / 响应体 / 错误码）
- 数据模型变更（字段 / 类型 / 默认值 / 迁移策略）
- 关键流程（复杂交互用伪代码或序列图）

**好的例子**：
```
### 接口设计
GET /api/posts/:id/related?limit=4
Request: path param id, query param limit (1-10, default 4)
Response 200: { items: [{id: string, title: string, coverUrl: string, publishedAt: string}], total: number }
Response 404: { error: 'post_not_found', postId: string }
Response 503: { error: 'compute_timeout', retryAfter: number }

### 数据模型
posts.relatedCache: JSON | null
结构：[{id, score}, ...] 最多 8 项（留 4 个兜底）
迁移：迁移脚本不回填（按需懒计算）；新文章发布时触发
失效：tag 变更时 hook 自动置 null
```

**差的例子**：
```
API 返回相关文章列表。数据库加个字段。  ← 无法照做
```

## 段落 5：工作量评估

**目标**：给出 P50 + P90 + 关键路径，方便排期和风险预警。

**好的例子**：
```
| 任务 | P50 | P90 | 依赖 |
|------|-----|-----|------|
| CMS: 字段 + 迁移脚本 | 0.5d | 1d | - |
| CMS: 相似度算法 | 1d | 2d | 字段 |
| CMS: API endpoint + 测试 | 0.5d | 1d | 算法 |
| 前端: 组件 + 样式 | 1d | 1.5d | - |
| 联调 + E2E | 0.5d | 1d | 全部 |
| **关键路径（P90）** | - | **5.5d** | - |

关键路径：CMS 字段 → 算法 → API → 联调。
前端组件可与 CMS 并行。
```

**差的例子**：
```
大概一周吧。  ← 无依赖分析，无 P90
```

## 段落 6：风险评估

**目标**：把方案层识别的风险（区别于 PRD 的业务风险）列出。

**好的例子**：
```
| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 冷启动慢：新文章首次请求需计算 | 中 | 单次首屏 +500ms | 异步计算，展示兜底 |
| tag 脏数据：部分文章 tag 为空 | 中 | 相似度为 0 | 上线前 SQL 扫描；空 tag 文章不展示模块 |
| 缓存击穿 | 低 | 计算并发 | 加内存锁，同 id 只计算一次 |
```

**差的例子**：
```
可能有性能问题。  ← 概率？影响？应对？
```

## 段落 7：回退方案

**目标**：让运维 / release 明确"出问题 10 分钟内能回到安全状态"。

**好的例子**：
```
回退策略：
- Level 1（10 秒）：feature flag `enable_related_posts=off`，用户立即看不到模块
- Level 2（5 分钟）：前端代码回滚 PR
- Level 3（30 分钟）：CMS API 回滚 PR（relatedCache 字段保留，不影响主流程）
- Level 4（不回退）：字段保留，不删除，避免迁移风险

回退 SLA：生产发现问题 → 10 分钟内 Level 1 → 2 小时内决定是否 Level 2
```

**差的例子**：
```
出问题就回滚。  ← 怎么回滚？多久？
```

## 段落 8：测试计划

**目标**：给 tester 交底"该测什么、测到什么程度"。细节交给 tester 执行。

**好的例子**：
```
### 单元测试
- 相似度算法：tag 完全重合 / 部分重合 / 无重合场景
- 时间衰减函数：近期文章权重 > 旧文章

### 集成测试
- API endpoint：正常 / 404 / 503 路径
- CMS hook：tag 变更触发 relatedCache 置 null

### E2E（Playwright）
- 详情页能看到相关阅读模块
- 点击相关文章跳转 + 埋点上报

由 tester 细化用例。
```

## 段落 9：安全与合规 checklist

严格按 SKILL.md 的清单过一遍，每条标"✅ 已考虑 / ❌ 不适用 / 🟡 需要进一步评估"。

**好的例子**：
```
### 安全
- [x] 输入校验：limit 参数上限校验（1-10）
- [x] 注入：使用 ORM，无原生 SQL
- [x] XSS：文章标题渲染前转义
- [x] 鉴权：相关阅读仅展示已发布文章
- [x] 速率限制：沿用现有 API gateway 的 rate limit

### 合规
- [x] PRD 无合规风险
- [x] 数据真实性：推荐基于真实文章 tag，无造数据
- [x] 外部平台：相关阅读是站内跳转，不涉及第三方
```

**差的例子**：
```
我审过了没问题。  ← 哪些条目？怎么审的？
```
