# 步骤 3：需求评审

## 前置条件

- 需求已录入需求池，拥有 record-id
- 需求状态 = "未启动"
- 任务目录下无 review.md

## 做什么

1. **深入理解需求**：
   - 阅读需求描述、PRD 文档
   - 理解业务背景和目标
   - 识别核心功能和边界

2. **合规与风险审查**（在输出评审文档前完成）：
   - 逐条审视 PRD 要求，识别是否存在**外部平台政策冲突**（如 Google 结构化数据指南、App Store 审核规则、第三方 API TOS）
   - 识别是否存在**数据真实性风险**（如伪造评分、虚构用户数据、捏造统计指标）
   - 发现风险时，在评审文档中明确标注并给出建议（保留/移除/替代方案），由用户决策

3. **输出评审文档**：
   - 创建 `review.md`
   - 内容应包含：
     - 需求来源（ID、描述、Owner、来源文档）
     - 需求范围（核心功能、边界条件、不包含）
     - 验收标准
     - 技术约束
     - 合规风险（如有，标注来源政策和违规后果）
     - 问题与风险
     - 评审结论

4. **更新 STATUS.md**（→ tripo-task-dirs）

5. **提议状态变更**：
   - 需求状态 → "定容确认"

6. **飞书通知用户**（→ tripo-tables，notification.md 节点 1）：
   通知后**暂停**，等待用户确认。

7. **等待用户确认容量**

## 如何定义完成

- [ ] 合规与风险审查已完成（无风险 或 风险已在 review.md 中标注）
- [ ] review.md 已输出
- [ ] STATUS.md 已更新
- [ ] 已提议状态变更为"定容确认"
- [ ] 用户已确认容量（进入下一步）
