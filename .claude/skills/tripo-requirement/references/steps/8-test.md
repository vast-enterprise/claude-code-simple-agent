# 步骤 8：自动化闭环

## 前置条件

- PR 已创建（所有涉及仓库）

## 做什么

> 子步骤严格顺序执行。每个子步骤声明了前置条件，前置条件未满足时**禁止**开始该子步骤。

### 8.1 自动化 Code Review

**前置条件**: 无（步骤 8 入口）

对**所有涉及仓库**执行代码审查（使用 `superpowers:requesting-code-review`）。

- 多仓库时可并行审查，但必须**全部审查完成**后才算 8.1 完成
- 发现 Critical/High 问题必须修复、推送、重新验证测试通过

**输出产物**（每个仓库各一份）：
- 路径：`tasks/$TASK_ID/code-review-<repo-name>.md`
- 内容：审查结果（按 CRITICAL/HIGH/MEDIUM/LOW 分级）、修复记录、最终测试结果
- 必须推送到对应仓库的 PR comment 或作为 commit 包含

**完成标志**: 所有仓库审查完成，审查报告已输出，所有 Critical/High 已修复并推送，测试通过。

### 8.2 编写测试计划

**前置条件**: 8.1 完成

输出集成测试计划文档：
- 路径：`tasks/$TASK_ID/integration-test-plan.md`
- 内容：测试环境、测试场景、前置条件、测试步骤

**完成标志**: 文档已写入。

### 8.3 集成测试

**前置条件**: 8.2 完成

**执行前检查清单**（禁止跳过）：
- [ ] 已读取测试计划，确认测试目标（测什么组件/功能）
- [ ] 测试数据覆盖组件全部 props/参数（不是只填一个）
- [ ] 确认验证指标：用户看的是什么？我测的是什么？

按测试计划逐项执行，分两类：

**API 测试**（后端 endpoint、数据写入）：
- 必须启动服务后用 curl / httpie 实际调用，贴请求和响应原文
- 禁止用"单元测试已覆盖"替代实际 API 调用

**UI 测试**（前端渲染、交互）：
- 使用 `playwright-cli` 技能执行浏览器集成测试
- 截图或 DOM snapshot 作为证据

发现问题则修复、推送、重新测试，直到通过。

**执行后检查清单**（禁止跳过）：
- [ ] 证据已贴（DOM 结构 / Network 请求 / Console 输出）
- [ ] 结论基于证据，不是基于假设或脑补

**降级条件**（仅以下情况允许跳过某项测试场景）：
- 服务无法启动（需贴启动失败日志）
- 测试场景依赖外部服务且不可 mock（需说明具体依赖）

降级的场景必须在报告中标注 `⚠️ DEFERRED`，说明原因和验收阶段的补测计划。

**完成标志**: 测试计划中每个场景状态为 ✅ PASS 或 ⚠️ DEFERRED（附原因），无 ❌ FAIL。

### 8.4 输出验证报告

**前置条件**: 8.3 完成

- 路径：`tasks/$TASK_ID/integration-test-report.md`
- 内容：测试环境、测试结果、修复的问题

**完成后**：更新执行表"计划提测时间"为当天实际日期，反映真实提测时间。

**完成标志**: 文档已写入，执行表"计划提测时间"已更新为实际日期。

### 8.5 飞书通知用户

**前置条件（硬性阻塞，全部满足才可执行）**:

- [ ] 8.1 所有仓库 Code Review 完成，问题已修复
- [ ] 8.2 测试计划文档已输出
- [ ] 8.3 集成测试已通过
- [ ] 8.4 验证报告已输出

**以上 checklist 必须逐项确认全部 ✅ 后才可发送通知。缺任何一项即禁止通知。**

参见 [notification.md](../notification.md) 节点 3。

通知后**暂停**，等待用户验收。

## 如何定义完成

- [ ] 所有仓库 Code Review 已执行，问题已修复
- [ ] 测试计划文档已输出
- [ ] 集成测试已执行，通过
- [ ] 验证报告已输出
- [ ] 已通知用户
