# 步骤 10：提交发车候选

> **重要**：本步骤**只负责把需求推到「发车候选」状态**,**不负责真正发车上线**。
> 发车是独立流程,由用户/scrum-master 在合适时机统一编排 → 见 **tripo-release**。

## 前置条件

- 用户验收通过(步骤 9 完成)
- 执行表状态 = "测试中",需求池状态 = "验收/提测中"

## 做什么

### 10.1 需求准出(入发车候选队列)

执行"需求准出",把需求从执行表搬进发车中需求表,排队等发车：

1. **确认准出路径**:跟车 / SSS / Hotfix(→ 见 tripo-release/references/dispatch-board.md「三条发车路径」语义)
   - 默认走跟车,SSS/Hotfix 需要用户显式指定
2. **提议 + AskUserQuestion 确认路径**,不替用户决策
3. **执行 lark-cli 操作**(option_id / 字段 → tripo-tables/references/release-flow.md):
   - 发车中需求:upsert 新记录,状态=**待上线**(跟车)或**上线中**(SSS/Hotfix),关联发车版本
   - 执行表:状态 → **完成**
   - 需求池:需求状态保持 **验收/提测中**(直到发版完成确认才变「已完成」)

### 10.2 更新任务目录

1. STATUS.md:标注「已提交发车候选,等待 scrum-master 统一发车」(→ tripo-task-dirs)
2. **不归档**——任务目录等发版完成确认后再归档

### 10.3 通知(可选)

- 按 tripo-notify 节点规则,可在发车群简短同步「XX 需求已入发车候选」
- **不主动触发发车**,等用户说「发车」或 scrum-master 汇总

## 不做什么

- ❌ **不调用 tripo-release 发 production**
- ❌ **不 tag、不触发 workflow、不勾部署 checkbox**
- ❌ **不把需求池状态改为「已完成」**(那是发版完成确认的职责)
- ❌ **不归档任务目录**(等车到站)

## 如何定义完成

- [ ] 发车中需求表已有本需求记录,状态=待上线(跟车)/ 上线中(SSS/Hotfix)
- [ ] 执行表状态 = 完成
- [ ] 需求池状态 = 验收/提测中(未变「已完成」)
- [ ] STATUS.md 标注「已提交发车候选」
- [ ] 任务目录**未归档**

## 本步骤结束后的状态

| 表格 | 状态字段 | 值 |
|------|---------|-----|
| 需求池 | 需求状态 | 验收/提测中 |
| 执行表 | 状态 | 完成 |
| 发车中需求 | 状态 | 待上线 / 上线中 |

需求池 → 已完成 / 发车中需求 → 完成 / 任务目录归档 发生在**发车流程结束**后(→ tripo-release Step 11-13,由 scrum-master 操作)。
