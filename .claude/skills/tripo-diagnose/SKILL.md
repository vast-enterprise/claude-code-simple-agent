# 问题诊断方法论

## 不跳级

诊断按风险从低到高分四级，从第 1 级开始，穷尽线索后再升级。

| 级别 | 做什么 | 门槛 |
|------|--------|------|
| 1 代码静态分析 | 读代码、对比同类实现、检查配置 | 自由使用 |
| 2 本地复现 | 启动服务、构造数据、触发问题 | 自由使用 |
| 3 Staging 验证 | 本地无法复现时用 staging 验证 | 自由使用 |
| 4 生产环境 | curl 生产 API、查数据库、注入脚本 | 🔴 必须 AskUserQuestion 获得确认 |

第 4 级即使是"只读"操作也需要审批——可能触发监控告警或暴露敏感数据。

"穷尽"不是"试了一下没找到"，是系统性地检查了所有相关文件和配置。

## 失败回退

同方向连续失败 2 次，必须换方向——换思路，不是换参数。

判断"同方向"：都在调同一类东西（不同的浏览器参数、不同的 API endpoint）就是同方向。从"调参数"切到"读源码找根因"是不同方向。

预期中的 RED 测试和已声明目的的探测性尝试不算失败。

## 假设驱动

基于现象提出假设，有针对性地验证。不要漫无目的地翻代码。

每个假设的结果都要记录，包括排除的——排除项是诊断的一部分，证明你系统性地检查过了。

## 影响范围

找到根因后，检查同类问题是否存在。一个 bug 暴露的往往是一个模式缺陷——修一个不够，要扫一类。

## 诊断产出

诊断结束后输出四项：

- **根因**：一句话说清问题本质
- **证据链**：从现象到根因的推导路径
- **影响范围**：同类问题是否存在
- **修复建议**：交给谁执行
