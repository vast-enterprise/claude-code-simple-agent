# 问题诊断方法论

## 一、诊断的优先级阶梯

> 🔴 **从代码开始，逐级升级，不跳级**

```
现象输入（bug 报告/截图/错误信息）
│
├── 第 1 级：代码静态分析（自由使用）
│   ├── 读相关代码文件
│   ├── 对比同类实现（如 blockquote.ts vs paragraph.ts）
│   ├── 检查注释与代码一致性
│   └── grep 配置文件确认设置
│
├── 第 2 级：本地复现（自由使用）
│   ├── 在本地 dev 环境启动服务
│   ├── 构造测试数据复现问题
│   └── 用 playwright/curl 验证
│
├── 第 3 级：Staging 验证（自由使用）
│   └── 如有 staging 环境，在 staging 上验证
│
└── 第 4 级：生产环境检查（需审批）
    └── 🔴 必须 AskUserQuestion 获得所有者确认后才能操作
        包括：curl 生产 API、连接生产数据库、在线上页面注入脚本
```

### 为什么不跳级

- 第 1 级能解决大多数问题，成本最低、风险为零
- 跳到第 4 级（生产环境）的风险：数据污染、安全暴露、用户可见的异常
- 实际案例：blockquote bug 通过对比 paragraph.ts 的写法就能定位，无需查生产数据

## 二、每级怎么做

### 第 1 级：代码静态分析

这是最重要的一级。目标：穷尽代码层面的线索后再进入下一级。

**核心方法：同类对比法**

```
1. 找到出问题的代码文件（如 blockquote.ts）
2. 找 3 个同类文件（如 paragraph.ts、heading.ts、list.ts）
3. 逐行对比，找差异：
   - 用了不同的函数？（h('blockquote') vs h(ProseBlockquote)）
   - 注释说一套，代码做另一套？
   - 缺少了某个注册/配置步骤？
```

**配置/环境问题的诊断**

```
1. grep -i '<关键词>' <配置文件>
   └── 如：grep -i 'https\|ssl\|cert' nuxt.config.ts
2. 理解配置的含义（看注释或文档）
3. 确认配置与当前行为是否一致
```

**认证错误（401/403）的诊断**

```
1. echo $VAR_NAME           → 确认 shell 实际值
2. grep VAR_NAME .env       → 确认 .env 期望值
3. 应用启动日志（logEnv）   → 确认运行时读到的值
三者不一致 → 系统变量优先级覆盖问题（dotenv 不覆盖已存在的系统变量）
```

### 第 2 级：本地复现

当第 1 级无法确定根因时，在本地复现。

```
1. 启动服务（→ tripo-dev 的 Pre-flight Check）
2. 构造测试数据（模拟 bug 报告中的场景）
3. 用 playwright 或 curl 触发问题
4. 检查 console、network、DOM 状态
```

### 第 3 级：Staging 验证

当本地无法复现（如数据差异、环境差异）时，使用 staging 环境。

### 第 4 级：生产环境检查

> 🔴 **必须先 AskUserQuestion 获得所有者确认**

即使是"只读"操作（curl、数据库 query、页面 JS 注入），也需要审批。原因：
- "只读"操作也可能触发日志/监控告警
- 生产数据敏感，不应在未授权情况下查看
- 所有者需要知道谁在什么时候访问了生产环境

## 三、失败回退机制

> 🔴 **同方向连续失败 2 次 → 必须换方向**

```
尝试方案 A → 失败
│
├── 分析失败原因，调整后重试方案 A → 还是失败
│
└── 🔴 STOP：不能继续在方案 A 上尝试
    ├── 回退一步，重新诊断
    ├── 列出至少 2 个替代方向
    └── 选择一个本质不同的方向继续
```

**什么算"同方向"**：
- 都在试不同的 chromium 参数 → 同方向
- 都在换不同的 API endpoint 查数据 → 同方向
- 从"试参数"切换到"读配置文件找根因" → 不同方向

**什么不算"失败"**：
- 预期中的 RED 测试 → 不算
- 故意的探测性尝试（已声明目的）→ 不算

## 四、什么是好的诊断

### 4.1 好诊断的三个特征

1. **有假设**：不是漫无目的地翻代码，而是基于现象提出假设再验证
   ```
   现象：线上 blockquote 不显示
   假设 1：转换器未注册 → 检查 default-converters.ts → 已注册，排除
   假设 2：渲染方式有问题 → 对比 paragraph.ts → 发现用了原生标签而非 Prose 组件
   ```

2. **有边界**：知道什么时候该停、什么时候该升级
   - 第 1 级分析了 30 分钟没结论 → 升级到第 2 级
   - 同方向连续 2 次失败 → 换方向

3. **有交代**：每一步的发现都记录，即使是排除项
   ```
   ✅ 转换器已注册（default-converters.ts:15）
   ✅ CSS 选择器存在（blog.css:42）
   ❌ 渲染方式异常：h('blockquote') 应为 h(ProseBlockquote)
   ```

### 4.2 诊断产出

诊断结束后应输出：
- **根因**：一句话说清楚问题本质
- **证据链**：从现象到根因的推导路径
- **影响范围**：同类问题是否存在（如果 blockquote 有 bug，list/table 是否也有？）
- **修复建议**：可以交给 tripo-bugfix 或 tripo-dev 执行

### 4.3 Handoff：诊断之后

```
诊断完成，根因已确认
│
├── 需要修复代码 → 加载 tripo-dev（编码方法论）
├── 需要走缺陷修复流程 → 加载 tripo-bugfix（修复流程）
└── 修复后需要验证 → 加载 tripo-test（测试方法论）
```
