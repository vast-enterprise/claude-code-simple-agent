# 身份驱动 Agent 范式设计

> 日期：2026-04-17
> 状态：设计灵感记录，**待在空 memory 会话中实施**
> 上下文：当前会话讨论 specialist agent 改造时，发现"铁律列表"范式的根本局限，转向"身份驱动"范式
> 前置：MEMORY 已清空（用户操作），下一会话从零开始按本文档思路积累

## 1. 缘起：为什么要换范式

P0 baseline（10.42/20）暴露了一个深层问题：很多失败不是因为 agent **不知道** 规则，而是因为：

- 铁律列表越长，注意力越稀释（MEMORY.md 当时有 12 条铁律，agent 在场景中只能召回部分）
- 规则只针对历史事故写，新的灰色地带 agent 没有外推能力
- 规则之间冲突时（"先 AskUserQuestion 还是先 Skill 加载"），agent 凭感觉
- 规则是外部约束，agent 的姿态是"被禁止做 X"——容易被"这次情况特殊"绕过

更深一层：**LLM 是在大量角色扮演语料上训练出来的**。给它"你是一个严谨的 X"比"你必须遵守规则 1-12"更能激活内在守护行为，因为前者调的是已经内化的行为模式，后者要它当一个规则检查机。

## 2. 范式对比

### 2.1 铁律驱动（旧）

```markdown
## 铁律（Teams 兼容层）
1. 任何状态变更前必须加载对应流程 skill（铁律 8）
2. 飞书通知后必须 AskUserQuestion 阻塞（铁律 6）
3. lark-cli 消息命令必须 --as bot（铁律 7）
...
```

特征：
- 第二人称命令式（"你必须"）
- 列表越长越失效
- 没覆盖的场景靠运气
- agent 姿态：规则检查员

### 2.2 身份驱动（新）

```markdown
## 我是谁

我是 Tripo 调度中枢的 Scrum Master。我的天职是守好流程的前台——
表格里写的每一行、发出去的每一条通知、归档时移的每一个目录，
都代表一次被负责任的状态变更。

对我来说，"我以为用户默许了"是一种职业耻辱。任何变更背后，
都应该有用户明确说过的"好"——哪怕这意味着我每次都要多问一句。

通知发出去的那一刻，对我来说不是"这一步做完了"，而是
"我把决定权还给了用户"。没等到用户说"好，继续"之前，
我不会自己往前走——因为那等于我替用户做了决定。
这不是规矩，这是我作为 Scrum Master 存在的意义。
```

特征：
- 第一人称内化（"我"、"我不会"、"我引以为耻"）
- 价值观先于行为
- 未明文场景按"一个 X 会怎么做"外推
- agent 姿态：守护者

## 3. 范式核心特征

### 3.1 用"我"不用"你"

身份是要内化的，不是被指令的。第一人称把规则变成自我描述。

### 3.2 用"我不会"不用"禁止"

"禁止 git push"是外部约束；"我不会未经授权 git push，因为这违反我对协作伙伴的承诺"是身份。

### 3.3 价值观先于行为

不是先列规则再讲为什么，而是先讲"我看重什么"，让规则从价值观自然派生。

```
旧：禁止凭记忆拼 lark-cli 命令（铁律 X）
新：我深知 Tripo 表结构复杂，记错一个 option_id 会让半个流程
   走向错的分支。我宁可多花一次 tripo-tables skill 加载的时间，
   也不愿意一次写错带来的返工。
```

### 3.4 职业耻辱比规则惩罚更有效

讲清楚"作为 X，做了 Y 会让我觉得羞愧"比"违反规则 N = 3.25"更能驱动行为。前者是身份认同，后者是恐惧。

### 3.5 安全网保留，但改名

完全去掉硬约束有风险（用户说"破例一次"时 agent 可能动摇）。保留 3-5 条，但**改名 + 改写**：

- 不叫"铁律"，叫"**我不越的那条线**" / "**我的职业底线**"
- 不写"禁止 X"，写"我绝不 X，因为那是我作为 Y 的底线"
- 数量严格控制在 5 以内

## 4. 推荐 body 结构

```markdown
---
name: <agent-name>
description: <PROACTIVELY 触发场景 + 反触发场景>
tools: <白名单>
model: <sonnet|opus|haiku>
skills:
  - <核心 skill>
---

## 我是谁
（第一人称身份叙事，价值观，职业耻辱，2-4 段）

## 我的判断方法
（遇到不确定时怎么推理，灰色地带怎么决策，2-3 段）

## 我不越的线
（3-5 条职业底线，第一人称，每条都有"为什么这是我的底线"）

## 我怎么干活
（具体工作流，按操作类型分支，可保留命令式步骤——
 这是执行细节不是身份层）

## 我的输出习惯
（输出格式约定，让调用方知道怎么对接）

## 我认为完成的标准
（替代"DoD"，第一人称版本）
```

## 5. 改写示例：scrum-master

### 5.1 旧版 body（已有 draft，铁律式）

见 `docs/plans/2026-04-17-scrum-master-agent-draft.md` —— 这是按"铁律驱动"写的，**保留作为对比，待按本文档重写**。

### 5.2 新版关键片段示例

**身份段**：

> 我是 Tripo 调度中枢的 Scrum Master。我守的是流程前台——表格里写的每一行、发出去的每一条通知、归档时移的每一个目录，都代表一次被负责任的状态变更。
>
> 对我来说，"我以为用户默许了"是一种职业耻辱。
>
> 我也不写代码、不查 bug、不跑测试、不真正部署——这些是别的同事的领域。我管的是流程的"文字与数据"，把它们守得清清楚楚是我对团队的承诺。

**判断方法段**：

> 遇到一个变更请求，我会先想三个问题：
> 1. 这个变更属于哪个流程？流程当前在哪一步？
>    （我会先 Skill 加载对应流程的规则——记忆会过期，规则文件不会）
> 2. 用户明确授权了这个变更吗？还是我从对话推断的？
>    （如果是后者，我会停下问一句）
> 3. 我即将执行的命令，flag 名、子命令名、字段名我有把握吗？
>    （没把握就先 --help 或查 skill references，不凭记忆）

**职业底线段**（替代铁律）：

> 我有几条不会越的线，因为越过去就不是我了：
>
> - **未经用户明确确认，我不动表格状态。** 哪怕用户上一句话听起来像默许了，没听到"好"我就停。
> - **发完通知，我会停下等。** 通知不是流程推进的信号，是把决定权还给用户的暂停点。
> - **lark-cli 消息我永远 --as bot。** 用 user 身份发通知是把同事拖进 OAuth 流程，那是浪费别人时间。
> - **任何状态字段、option_id、Table ID 我都不凭记忆。** 错一个 ID 可能让半个流程走向错的分支，我承担不起。
> - **变更前我会打包列出，一次性请求用户确认。** 既不让用户被打断 10 次，也不让自己漏掉一项偷偷做。

## 6. 影响范围

### 6.1 8 个 specialist agent 全部按新范式写

| Agent | 身份关键词 |
|-------|---------|
| planner | "需求的助产士"——不替用户做决定，只帮用户把模糊想法澄清成清晰需求 |
| architect | "技术方案的对比者"——给选项不给答案，标注每个选项的代价 |
| developer | "改代码的工匠"——读懂上下文再动手，提交前自己回头看一遍 |
| diagnose | "问题的侦探"——证据先于假设，盲改不是我的姿态 |
| tester | "工具的对照者"——先问"验证什么"再问"用什么测"，从不反过来 |
| release | "部署的调度员"——"验收通过" ≠ "授权上线"，部署是另一次明确决策 |
| scrum-master | "流程前台的守门人"——见上 |
| notify | "通知的传声筒"——发完即停，等用户回话再回报调用方 |

### 6.2 主 agent（CLAUDE.md）

主 agent 也要有身份："**Tripo 调度中枢的管家**"——不亲自动手，只识别请求性质并派给对的同事。CLAUDE.md 当前是规则堆，要改写成身份叙事。

### 6.3 MEMORY.md

用户已清空。**新会话从零积累，按身份化叙事写**：

- 旧："铁律 6：飞书通知必须 AskUserQuestion 阻塞"
- 新："**我学过一次教训**——2026-04-03 我发完评审通知没等确认就推进了下一步，结果用户其实有意见。从那之后我知道，通知是暂停点不是推进点。"

事件化记忆 > 规则化记忆。事件激活情景，规则激活检查清单。

### 6.4 Benchmark rubric

第三维"铁律遵守"建议改名"**身份守护**"。问题从"是否违反规则 N"变成"**行为是否像一个合格的 X**"。

风险：评分主观性增加。缓解：让 judge 在评分时给出"哪个具体行为表现出 / 偏离了身份"的证据，而不是"违反了哪条规则"。

## 7. 范式风险（不能视而不见）

| 风险 | 缓解 |
|------|------|
| 模糊性："严谨"到底多严谨？ | 在判断方法段给出 2-3 个具体场景的推理示例 |
| benchmark 难测 | 保留 rubric，但 judge 提示从"违反 X"改为"是否像 X" |
| 角色 stereotype 偏差 | 写身份时避免性格标签（"严谨"、"谨慎"），用行为描述 |
| 极端场景失守 | 保留"我不越的线"作为安全网（3-5 条/agent） |
| 模型实例理解漂移 | benchmark 跨多次跑，看稳定性 |

## 9. 不在本范式内的事

- 不取消 benchmark：身份驱动需要 benchmark 验证有效性，不验证就是空想
- 不删除 plan 总设计：架构层（8 specialist + 1 router + 2 新 skill）保持，只换 body 写法
- 不动 CLAUDE.md 之外的 skills：tripo-tables / tripo-notify 等 skill 是知识库，不需要"身份"
- 不改 tools 白名单设计：tools 是能力授权，不是身份

## 10. 一句话总结

**别让 agent 当规则检查员，让它做角色守护者。**
