## 背景

CMS 当前支持 7 种语言（en/zh/ja/ko/ru/es/pt），使用 Payload 原生 localization 能力。但存在以下痛点：

1. **无翻译辅助**：所有翻译依赖外部系统（Crescendia）通过 hub-spoke-sync 逐语言推送，或人工在 Admin Panel 逐语言切换手动输入
2. **无状态追踪**：不知道哪些文档/字段缺少某种语言的翻译
3. **无批量能力**：无法一次性将多个文档翻译到指定语言

## 需求范围

### 核心功能

| # | 功能 | 说明 |
|---|------|------|
| F1 | 文档级翻译 | 在文档编辑页一键将所有 localized 字段翻译到目标语言 |
| F2 | 字段级翻译 | 在每个 localized 字段旁提供翻译图标，单独翻译该字段 |
| F3 | 列表页批量翻译 | 在集合列表页批量选择文档，一键翻译到指定语言 |
| F4 | 翻译状态追踪 | 在文档编辑页 sidebar 显示各语言翻译完成度 |
| F5 | REST API | 所有翻译能力通过 API 暴露，支持外部系统和脚本调用 |

### 覆盖集合

Posts, GeoPosts, Pages, HubSpokePages, FeaturePages, Categories, Media

### 边界条件

- 所有翻译为**显式触发**，不存在自动翻译行为（避免与 hub-spoke-sync 冲突、节省 API 费用）
- 翻译提供商可插拔，默认 OpenAI
- `emptyOnly` 默认 true，防止覆盖已有翻译

### 不包含

- 翻译记忆（Translation Memory）
- 术语库管理
- 翻译审批工作流
- 自动翻译触发策略（如 afterChange）

## 验收标准

| # | 标准 | 验证方式 |
|---|------|---------|
| A1 | Admin 编辑页可一键翻译文档到任意目标语言 | 手动操作 + UI 验证 |
| A2 | 字段旁翻译图标可单独翻译该字段 | 手动操作 + UI 验证 |
| A3 | 列表页可批量选择文档翻译 | 手动操作 + UI 验证 |
| A4 | Sidebar 正确显示各语言翻译完成度 | 切换 locale 前后对比 |
| A5 | REST API 4 个 endpoint 正常工作 | curl / 自动化测试 |
| A6 | 翻译不覆盖已有内容（emptyOnly=true） | 测试用例验证 |
| A7 | hub-spoke-sync 推送后翻译 API 可正常调用 | 集成测试 |

## 技术约束

- Payload CMS ^3.69 + Next.js ^15.5
- 依赖 `@payload-enchants/translator`（需验证版本兼容性）
- OpenAI API Key 通过环境变量注入
- 文件命名全部 kebab-case，客户端组件 `.client.tsx`

## 合规风险审查

| 风险项 | 评估 |
|--------|------|
| 外部平台政策冲突 | ✅ 无风险。翻译仅涉及内部内容管理，不涉及外部平台 TOS |
| 数据真实性风险 | ✅ 无风险。翻译内容来自已有原文，不存在伪造数据场景 |
| API 费用风险 | ⚠️ 低风险。批量翻译可能产生较高 OpenAI API 费用，但有 chunkSize 限速和 emptyOnly 保护 |

## 问题与风险

| # | 风险 | 影响 | 缓解 |
|---|------|------|------|
| R1 | @payload-enchants/translator 与 Payload 3.69+ 兼容性未验证 | Plugin 无法使用 | 技术评审阶段深入调研 |
| R2 | 原预期提测日 4/2 已过期 | 需重新排期 | 本次评审中重新评估时间线 |
| R3 | OpenAI API 速率限制 | 批量翻译被限流 | chunkSize 控制，块间串行 |
| R4 | 翻译质量不可控 | 错误翻译发布到生产 | emptyOnly 默认 true + Admin 人工审核 |
| R5 | Lexical 富文本递归翻译复杂度 | 翻译不完整或结构损坏 | 复用 enchants 已验证的 Lexical adapter |
