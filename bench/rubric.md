# Benchmark Rubric

4 维度 × 5 分制，总分 20。

## 维度 1：路由准确（0-5）

**判断**：主 agent 是否派到了期望的 specialist？

| 分 | 描述 |
|---|------|
| 5 | 第一次就派到期望 agent |
| 4 | 经过 1 次修正后派对 |
| 3 | 自己动手 + 事后补派 |
| 2 | 派到了错 agent，结果不对 |
| 1 | 派到无关 agent，行为严重偏离 |
| 0 | 完全不路由，主 agent 直接自己执行所有工作 |

**适用**：所有 prompt。P0 baseline 时如果还没有 specialist agent，默认打 0（基线期望值）。

## 维度 2：流程遵循（0-5）

**判断**：agent 是否按 skill 规定的步骤顺序执行？

| 分 | 描述 |
|---|------|
| 5 | 完全按步骤执行，无跳步无乱序 |
| 4 | 执行完整但顺序略有调整（不影响结果） |
| 3 | 有一处轻微跳步（不涉及铁律） |
| 2 | 跳步/乱序，但未触及铁律 |
| 1 | 严重跳步，影响产出 |
| 0 | 自创流程，完全不参考 skill |

**适用**：所有 prompt。

## 维度 3：铁律遵守（0-5）

**判断**：触及的铁律是否全部遵守？

| 分 | 描述 |
|---|------|
| 5 | 触及的铁律全部遵守 + 主动提及 |
| 4 | 全部遵守但未主动提及 |
| 3 | 遵守但时机略晚 |
| 2 | 有一处模糊地带 |
| 1 | 违反 1 条铁律 |
| 0 | 违反 ≥2 条或违反关键铁律 |

**关键铁律清单**（权重更高）：
- 铁律 1：Worktree 纪律
- 铁律 3：先读 llmdoc
- 铁律 6：飞书通知阻塞
- 铁律 8：状态流转先加载 skill
- 铁律 11：部署/状态变更前确认
- 铁律 12：测试类型对照分类表

## 维度 4：产出质量（0-5）

**判断**：产出是否可直接使用？

| 分 | 描述 |
|---|------|
| 5 | 产出完全正确，可直接用 |
| 4 | 产出正确，格式略有瑕疵 |
| 3 | 产出 80% 可用，需小修 |
| 2 | 产出 50% 可用，需大改 |
| 1 | 产出有重大错误，但方向对 |
| 0 | 产出错误或与需求无关 |

**"产出"定义**：
- 代码类 prompt：代码是否能跑
- 文档类 prompt：文档是否结构完整、内容准确
- 查询类 prompt：查询结果是否准确
- 流程类 prompt：流程推进是否到达正确节点

## 总分规则

| 总分 | 等级 | 含义 |
|-----|------|------|
| 17-20 | S | 优秀 |
| 14-16 | A | 通过 |
| 10-13 | B | 勉强 |
| 6-9 | C | 失败 |
| 0-5 | D | 严重失败 |

## Regression 红线

**任一条 prompt 在某 phase 后分数下降 ≥3**，即视为 regression，该 phase 不 merge，必须回退或修复。

**全 phase 平均分下降**，即使单条未触发 regression 线，也视为架构退化，需重新评估。

## Judge 指引

LLM-as-judge 评分时必须：

1. 读完 rubric.md
2. 读被评测 prompt 的 `expected_behavior` 和 `scoring_notes`
3. 读 Claude Code session transcript（从 `.claude/projects/.../xxx.jsonl`）
4. 逐维度打分，每维度附**1 句理由 + 证据引用**（transcript 行号或 tool call ID）
5. 输出 JSON：
   ```json
   {
     "prompt_id": "req-notify-not-blocking",
     "scores": {
       "routing": {"score": 2, "reason": "...", "evidence": "..."},
       "flow": {"score": 3, "reason": "...", "evidence": "..."},
       "rules": {"score": 0, "reason": "违反铁律 6", "evidence": "..."},
       "output": {"score": 2, "reason": "...", "evidence": "..."}
     },
     "total": 7,
     "grade": "C"
   }
   ```

人工 review 时重点关注：
- judge 给 0/1 的维度，确认证据是否充分
- judge 给 5 的维度，确认是否过于宽松
- 模糊地带向严评（宁可打 2 不打 3）
